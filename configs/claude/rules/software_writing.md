# Software Writing Rule

## 命名

- 汎用的な名前 (`value`, `data`, `result`) を避け、具体的な意味を持つ名前を付ける
- 動詞は動作を正確に表すものを選ぶ (`get` より `fetch`, `download`, `calculate` など)
- 変数名のスコープが広いほど説明的な名前を、狭いほど短い名前を使う
- 単位や文脈が曖昧な場合はサフィックスで補足する (`duration_ms`, `price_yen`)
- ドメインのユビキタス言語を名前に反映し、コードと業務用語を一致させる
- 否定形の名前 (`is_not_valid`) を避け、肯定形 (`is_valid`) を使う

## コメント

- コードから読み取れない「なぜ (Why)」を書き、「何をしているか (What)」は書かない
- 設計判断の背景、制約、トレードオフをコメントで残す
- 自明な処理にコメントを付けない
- TODO, FIXME, HACK, XXX 等の標準マーカーを活用し、未解決事項を明示する
- 公開インターフェースには目的・引数の制約・戻り値の意味を記述する

## 制御フロー

- ガード節と早期リターンでネストを浅く保つ
- 条件式は肯定形を優先し、否定条件のネストを避ける
- 比較は自然言語の語順に近い形で書く (`if age >= 18` > `if 18 <= age`)
- 複雑な条件式は説明変数に抽出して意図を明示する
- 三項演算子は単純な式に限定し、ネストさせない

## 型と不変性

- 型・アノテーションを積極的に付与し、コードの意図を型で伝える
- 意味的に異なる値は別の型で区別する (例: `UserId` と `OrderId` を `int` で混同しない)
- 不正な状態を型で表現不可能にする (Make Illegal States Unrepresentable)
- データは不変 (immutable) をデフォルトとし、可変が必要な場合のみ明示する
- Option/Result 型を活用し、null や例外による暗黙のエラー伝播を避ける

## 関数設計

- 関数は単一の責務を持ち、名前が示す処理だけを行う
- 純粋関数を優先し、副作用はシステム境界に追い出す
- 引数の数は少なく保つ (多い場合は構造体やオブジェクトにまとめる)
- 無関係な処理を1つの関数にまとめず、小さな関数に分割して合成する
- 戻り値の型で成功・失敗を表現し、呼び出し側にハンドリングを強制する
