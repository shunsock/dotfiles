# Software Design Rule

## 凝集度と結合度

- モジュールは単一の責務に集中させる (高凝集)
- モジュール間の依存は最小限にする (疎結合)
- 関連するデータと振る舞いは同じモジュールにまとめる
- 無関係な機能を1つのモジュールに混在させない

## 宣言的・存在的・意味的な構造

- 命令的な手続きより宣言的な定義を優先する
- コードの構造自体が意図を表現するように設計する
- 名前（変数名、関数名、型名）はドメインの意味を反映させる
- 「何をするか (What)」を表現し、「どうやるか (How)」は抽象化する

## 変更容易性とメンテナンス性

- 変更の影響範囲が局所化される設計にする
- 既存コードを変更せずに機能を拡張できる構造を目指す (開放閉鎖原則)
- 具体的な実装ではなく抽象に依存する (依存性逆転)
- 設定値や環境依存の値は外部から注入する

## 必要十分なコード量

- 不要な抽象化・過剰なユーティリティを作らない
- 同時に、必要な表現は省略しない
- 一度しか使わない処理を無理に共通化しない
- 3回以上繰り返される類似コードは共通化を検討する

## 可読性

- 型・アノテーションを付与し、コードの意図を型で伝える
- コードから読み取りにくい意図・背景・制約は補足コメントで説明する
- 自明な処理にはコメントを付けない
- 関数やモジュールの公開インターフェースには目的と制約を記述する

## 依存グラフのシンプルさ

- 循環依存を禁止する
- 依存方向を一貫させる (上位層から下位層への一方向)
- レイヤー間の境界を明確にし、飛び越えたアクセスを避ける
- 依存関係が複雑になった場合はモジュール分割を検討する
