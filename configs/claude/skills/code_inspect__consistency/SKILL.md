## 観点の定義

変更コードが、同一モジュール・隣接ファイル・プロジェクト全体の既存パターンと
一貫しているかを評価する。一貫性が低いコードは認知負荷を上げ、保守性を損なう。

## 評価手順

### Step 1: 既存コードの確認

変更ファイルと同じディレクトリ・モジュール内の既存ファイルを Read し、
プロジェクトで使われているスタイル・パターン・命名・ドメイン用語を把握する。

ドメイン用語の参照先を確認する。

```bash
ls .claude/rules/ubiquitous_language.md 2>/dev/null
```

存在すればこれを Read し、ユビキタス言語と「避けるべき表記」を把握する。

### Step 2: コーディングスタイルの一貫性 (定性)

- インデント・空白・改行が既存コードと揃っているか
- 文字列のクオート (シングル / ダブル) が統一されているか
- import 文の順序・グルーピングが既存パターンと一致しているか
- ファイル冒頭のコメント・ヘッダー規約に従っているか

軽微なスタイル差異 (フォーマッタで自動修正可能なもの) は nit で扱う。

### Step 3: 設計パターンの一貫性 (定性)

- エラーハンドリングの方式が既存コードと揃っているか (Result 型 vs 例外、ログ vs raise)
- ログ出力の形式・レベルが既存パターンと一致しているか
- 設定値の注入方法が既存と揃っているか (環境変数 / 設定ファイル / DI)
- データモデルの定義方法が既存と一致しているか (dataclass / Pydantic / TypedDict)
- テスト構造 (Arrange-Act-Assert / Given-When-Then) が既存と揃っているか

### Step 4: ドメイン用語の一貫性 (定性)

- ユビキタス言語に定義された用語を使っているか
- 「避けるべき表記」に該当する用語が混入していないか
- 同一概念に複数の名前 (例: `user_id` と `account_id` が混在) が使われていないか

### 判定基準

- ドメイン用語の誤用 (ユビキタス言語違反): must
- 既存パターンと明確に異なる設計選択: should
- 軽微なスタイル差異: nit

`.claude/skills/code_inspect/template/inspect_output.md` の規約に従う。
観点名は `consistency`。
