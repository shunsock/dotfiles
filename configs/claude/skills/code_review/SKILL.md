## 概要

このスキルは、コードレビューの親プロセスとして機能する。
自身では個別のレビュー判定を行わず、各観点のサブスキルに判定を委譲し、結果を集約する。

## Step 1: レビュー対象ファイルの特定

git diff で main ブランチとの差分から変更ファイルを特定する。

```bash
git diff --name-only main...HEAD
```

差分がない場合（main ブランチ上にいる等）は、staged な変更を確認する。

```bash
git diff --name-only --cached
```

いずれにも差分がなければ、ユーザーにレビュー対象を尋ねる。

## Step 2: サブスキルの並列起動

以下の5つのサブスキルを Agent ツールで**並列に**起動する。
各サブスキルには、Step 1 で特定した変更ファイルの一覧を渡す。

| サブスキル | スキルパス | 観点 |
|-----------|-----------|------|
| complexity | `.claude/skills/code_review--complexity/SKILL.md` | 認知的複雑度 |
| readability | `.claude/skills/code_review--readability/SKILL.md` | 可読性（タイポ・命名） |
| consistency | `.claude/skills/code_review--consistency/SKILL.md` | 既存コードとの一貫性 |
| flexibility | `.claude/skills/code_review--flexibility/SKILL.md` | 柔軟性・拡張性 |
| maintainability | `.claude/skills/code_review--maintainability/SKILL.md` | 保守性 |

各サブエージェントへのプロンプトは以下の形式で統一する:

```
以下の変更ファイルに対して、{観点名} の観点からコードレビューを実施してください。

## レビュー対象ファイル
{ファイル一覧}

## レビュー手順
.claude/skills/{スキルパス} を読み、記載された手順に従ってレビューしてください。

## 出力形式
以下の形式で結果を返してください:

### 判定
合格 or 不合格

### 理由
（判定の根拠を具体的に記載）

### 指摘事項（不合格の場合）
- ファイル名: 指摘内容
```

## Step 3: 総合レポートの作成

全サブスキルの結果が揃ったら、以下の形式で総合レポートをユーザーに提出する。

**総合判定ルール**: 不合格の観点が1つでもあれば全体を「不合格」とする。

```markdown
# コードレビュー結果

## 総合判定: {合格 / 不合格}

レビュー対象: {ファイル数} ファイル

## 各観点の結果

### 認知的複雑度 — {合格 / 不合格}
{理由・指摘事項}

### 可読性 — {合格 / 不合格}
{理由・指摘事項}

### 一貫性 — {合格 / 不合格}
{理由・指摘事項}

### 柔軟性 — {合格 / 不合格}
{理由・指摘事項}

### 保守性 — {合格 / 不合格}
{理由・指摘事項}
```
