# code_inspect__<観点> 出力フォーマット

このファイルは code_inspect__* サブスキルの出力契約を定義する。
各サブスキルはこのフォーマットに厳密に従って結果を返すこと。

## 出力契約

必ず以下の Markdown 形式で結果を返す。

```markdown
### 観点: <観点名>

### 所見
（観点全体に関する 2〜3 行の所見。CLI 不在等のスキップ事由もここに記載）

### 指摘
- [must|should|nit] <ファイル>:<行> — <何が問題か>
  - 理由: <なぜ問題か>
  - 代替案: <筋の良い修正案>

（指摘がない場合は "指摘なし" と記載する）
```

## 重要度の定義

- **must**: 修正必須。バグ・セキュリティ脆弱性・規約違反などの重大な問題
- **should**: 修正推奨。保守性・可読性・設計上の問題
- **nit**: 任意。好みのレベル・小さな改善・スタイル指摘

## 出力例

### 指摘がある場合

```markdown
### 観点: readability

### 所見
全体として命名は明確だが、一部に汎用的な変数名が残っている。

### 指摘
- [should] src/parser.py:42 — 変数名 `tmp` が汎用的で意図が伝わらない
  - 理由: 他の人が読んだときに何の一時変数か判断できない
  - 代替案: `parsed_user_input` のように意味を持つ名前に変更
- [nit] src/parser.py:88 — コメントが処理内容の繰り返しになっている
  - 理由: コードを読めば分かる内容で情報量がない
  - 代替案: 削除するか、なぜこの処理をしているかを書く
```

### 指摘がない場合

```markdown
### 観点: security

### 所見
今回の変更にセキュリティ上の懸念は見当たらない。

### 指摘
指摘なし
```

### CLI が利用できない場合

```markdown
### 観点: complexity

### 所見
complexity CLI が利用できないため、認知的複雑度の定量チェックをスキップしました。
`nix run nixpkgs#complexity` で実行可能です。定性評価のみで以下の指摘を行います。

### 指摘
- [should] src/handler.py:15 — 関数 `process_request` のネストが 4 段以上深い
  - 理由: 早期リターンで簡略化できる構造になっている
  - 代替案: バリデーション失敗時にガード節で return する
```
