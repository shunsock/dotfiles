## 観点の定義

変更コードが予期せぬ入力・例外・障害に対して適切に振る舞うかを評価する。
エラーハンドリングの不備は、デバッグ困難なバグの温床となる。

## 評価手順

### Step 1: エラー伝播の方式 (定性)

- エラーが Result 型 / Option 型 / 例外のいずれで伝播されているか、プロジェクトの方針と一致しているか
- 戻り値で成功・失敗を表現すべき箇所で例外が使われていないか (またはその逆)
- null / None / undefined の伝播が型システムで明示されているか
- 失敗が呼び出し側に強制的にハンドリングされる設計か (Result 型 / checked exception)

### Step 2: 握り潰しの有無 (定性)

- 空の except / catch ブロックがないか
- `except Exception: pass` のような汎用例外の握り潰しがないか
- 例外をログだけ出して握り潰し、上位に伝播させない箇所がないか
- エラーが意図的に握り潰される場合、その理由がコメントで説明されているか

### Step 3: エラーメッセージの十分性 (定性)

- エラーメッセージが原因の特定に十分な情報 (発生箇所・入力値・期待値) を含んでいるか
- ユーザー向けメッセージと開発者向けメッセージが分離されているか
- メッセージが内部実装の詳細を漏出させていないか (security 観点とも連動)
- スタックトレース / context が失われずにラップされているか

### Step 4: 例外の境界 (定性)

- 例外をキャッチする層が適切か (ビジネスロジックの中核ではなくシステム境界で)
- リソースのクリーンアップが finally / context manager / defer で確実に行われるか
- リトライ / フォールバック / circuit breaker が必要箇所に実装されているか
- パニック・ unrecoverable error と通常のエラーが区別されているか

### Step 5: バリデーション

- 外部入力 (ユーザー入力 / API 応答 / ファイル内容) が境界でバリデートされているか
- 内部関数の事前条件が型システムで担保されているか (バリデーションの過剰実装を避ける)
- バリデーションエラーが構造化されて伝播されているか

### 判定基準

- 例外の握り潰し・エラーが上位に伝わらない: must
- エラーメッセージが原因特定に不十分・リソースリーク: should
- エラー伝播方式の不統一 (consistency 観点とも重複しうる): should
- メッセージ表現の改善余地: nit
