## 観点の定義

変更コードがテスト可能な構造になっているかを評価する。
テスト容易性が低いコードはバグの混入リスクが高く、リファクタも困難になる。

## 評価手順

### Step 1: 副作用の境界 (定性)

- 副作用 (DB / API / ファイル IO / ネットワーク / 標準入出力) がシステム境界に追い出されているか
- ビジネスロジックの中核が純粋関数として抽出されているか
- 関数の中で副作用と計算が混在していないか

### Step 2: 純粋関数の活用 (定性)

- 同じ入力に対して常に同じ出力を返す関数が優先されているか
- 入力を変更せず新しい値を返すスタイル (immutable transformation) になっているか
- 不変オブジェクト (frozen dataclass / namedtuple / readonly) が活用されているか

### Step 3: 依存注入と差し替え可能性 (定性)

- 外部依存 (DB クライアント / API クライアント / time.now / random) が引数 / コンストラクタで注入可能か
- 直接インスタンス化されたクラスがテスト時にモックできない構造になっていないか
- 抽象 (interface / protocol) を介した依存になっているか

### Step 4: グローバル状態への依存 (定性)

- グローバル変数を読み書きしていないか
- シングルトン経由で状態を持ち回していないか
- 環境変数を関数内で直接読まず、設定オブジェクトを受け取る形になっているか

### Step 5: テストの追加 (定性)

- 変更内容に応じたテストが追加・修正されているか
- 既存テストが変更後も通る構造になっているか
- バグ修正の場合、再発防止のテストが追加されているか

変更規模に応じて判断する。設定値の追加・1 行の修正に対してテストを必須としない。

### 判定基準

- 副作用とロジックの混在によりテスト不可能な構造: must
- 依存注入されておらずモック困難・グローバル状態への依存: should
- テストが追加されるべき変更でテストが欠落: should
- 純粋関数化や不変オブジェクト化の改善余地: nit
