## 観点の定義

変更コードが、変更容易性と拡張性を備えた設計になっているかを評価する。
小規模な変更に対して過度な設計を求めない。変更の規模に見合った判断を行う。

## 評価手順

### Step 1: 依存性逆転 (定性)

- 具体的な実装ではなく抽象 (interface / protocol / ABC / trait) に依存しているか
- 外部サービスやインフラの詳細がビジネスロジックに漏れ出していないか
- 設定値・環境依存の値が外部から注入されているか (ハードコードがないか)
- マジックナンバー・固定文字列が定数として抽出されているか

### Step 2: モジュール境界 (定性)

- モジュールの公開インターフェースが最小限か
- 内部実装の詳細が外部に露出していないか (private/internal の活用)
- レイヤー間の境界が明確か (上位層から下位層への一方向依存)
- 循環依存が発生していないか
- レイヤーを飛び越えたアクセスがないか

### Step 3: 凝集度と結合度 (定性)

- モジュールが単一の責務に集中しているか (高凝集)
- 関連するデータと振る舞いが同じモジュールにまとまっているか
- 無関係な機能が 1 つのモジュールに混在していないか
- 他モジュールへの import が過度に多くないか

### Step 4: 拡張性 (定性)

- 新機能を追加する際に既存コードを修正せず拡張できる構造か (開放閉鎖原則)
- 条件分岐の追加ではなく、ポリモーフィズム / 戦略パターンで対応できる設計か
- 変更の影響範囲が局所化されているか
- 1 つの変更が広範囲のファイルに波及する構造になっていないか

### 判定基準

- 循環依存・ビジネスロジックへのインフラ詳細の漏出: must
- 抽象に依存すべき箇所での具象依存・モジュール責務の混在: should
- 軽微な改善余地・小規模変更に対する設計上の指摘: nit

小規模な変更 (設定値の追加、1 行の修正等) に対して過度な設計を求めない。
