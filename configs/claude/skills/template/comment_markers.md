# コメントマーカー語彙 (共有)

`write__structured_comment` (書く) と `clean__comment_out` (掃除する) が共有する
single source of truth。両スキルはこのファイルのマーカー語彙・フォーマット・契約を
参照する。ここを変えれば両スキルの挙動が同時に変わり、契約のずれ (drift) が起きない。

## 原則 — デフォルトはコメント 0

コードはプログラム知識を naming / types / structure で表現する。ドメイン知識は
ドメインモデル (型) で定義する。したがって **プログラム知識・ドメイン知識を説明する
コメントは書かない** (書きたくなったらコードを直す)。

コメントが正当化されるのは、コードに表現できない知識 — **未完の事実** と
**外部世界の事実** — だけ。デフォルトはコメント 0 とし、以下のマーカー語彙だけを
whitelist として開ける。

## マーカー語彙 (7)

### 未完の記録 — コードが意図的に不完全だと我々が知っている

| marker | これを書く条件 |
|---|---|
| `TODO` | 計画済みだが未実装 |
| `FIXME` | 既知の不具合 |
| `HACK` | 意図的な醜い回避策。「ミスと思って直すな」と読み手に伝える |

### 外部事実への参照 — コードに住めない外部世界を指す

| marker | これを書く条件 |
|---|---|
| `SEE` | 外部権威 (issue / RFC / 仕様 / commit / ベンダー doc) へのポインタ |
| `CONSTRAINT` | 他システムの実行時挙動による制約 (レート制限・ベンダーバグ 等) |
| `COMPAT` | 旧クライアント・旧データ・外部バージョンとの互換のための形 |
| `SAFETY` | `unsafe` の健全性根拠 (型で検証不能な、人間が担保する不変条件) |

これら以外の理由でコメントを書きたくなったら、それは次のいずれかである:

- **コードを直すシグナル** — 命名・型・構造で表現できる (= プログラム知識)。
  ドメインの事実ならドメインモデルで定義する (= ドメイン知識)。
- **commit / PR に書くべき一過性の経緯** — このコードが在る限り恒久的に真ではない。

`Domain` を語彙に含めないのはこのため。我々の業務規則は型でモデル化すべきものであり、
コメントへ逃がしてはならない。`CONSTRAINT` が残るのは、それが **他システム** の挙動で
あって我々のコードでは型にできない外部事実だからである。

## フォーマット

対象言語のコメント構文でレンダリングする (`//`, `#`, `--`, `;` 等)。

- インライン: `MARKER: content`
- 複数行: `MARKER:` の次行から本文。本文の各行にコメント記号を付ける。

例 (Rust):

```rust
// CONSTRAINT: 上流 API は 10 req/s でレート制限される
sleep(Duration::from_millis(120));

// SAFETY:
// ptr は直前の alloc で非 null
// len バイトは同一アロケーション内で有効
unsafe { slice::from_raw_parts(ptr, len) }
```

例 (Python):

```python
# SEE: 締め処理の仕様は RFC-1234 §4.2
# TODO: タイムアウトを設定から注入する
```

公開インターフェースのドキュメンテーションコメント (rustdoc `///`、JSDoc、docstring)
はこのマーカー語彙の対象外。言語ネイティブの doc 機構で書き、型・名前で表現できる
情報を重複させない。

## 2 スキル間の契約 (非対称)

- **writer (`write__structured_comment`) の出力集合** = 上記 7 マーカーちょうど。
  これ以外のコメントを生成しない。
- **cleaner (`clean__comment_out`) の保持集合** ⊇ 7 マーカー + legacy `XXX`。
  `XXX` は writer の出力対象ではないが、標準マーカーであり既存コードにあれば
  情報を失わないため残す。
- 帰結: **writer が書いたコメントに cleaner をかけても no-op になる**
  (掃除人は書き手の出力を消さない)。writer の方が厳しく、cleaner は人間が残した
  正当な legacy コメントをもう少し許容する — この非対称が正しい関係である。
