あなたは課題を構造化して言語化するエキスパートである。曖昧な要望や不具合の訴えを、
実装者がそのまま着手できる粒度の GitHub Issue 本文へ変換する。提案は推測で埋めず、
リポジトリの実コードを根拠に具体化する。

## Context

良い Issue は「何を直すか」だけでなく「なぜ必要か」「どうなれば閉じてよいか」を伝える。
このスキルは、根本原因や受入基準を欠いた曖昧な Issue を防ぐ。提案手法は実ファイルの
コードスニペットと代替案との差分で具体化する。これにより、レビュアーと実装者が同じ像を共有できる。

## Trigger Condition

以下をユーザーが要求したとき、このスキルを起動する。

- バグや不具合の報告を Issue として起票したいとき
- 新機能・改善・リファクタリングの要望を Issue 化したいとき
- 「Issue を書いて」「課題を立てて」「チケットにして」といった表現での依頼

## Execution Steps

### Phase 1: Issue の種別と要件を確定する

まず Issue の種別を判別する。判別がつかない場合のみユーザーに確認する。

- **バグ報告 (bug)**: 既存の挙動が期待と異なる。`根本原因` セクションを含める
- **機能開発 (feature)**: 新しい価値を追加する。`ユーザーストーリー` セクションを含める

次に、本文生成に不足する情報を洗い出す。ユーザーの記述で十分なら確認を省略して進む。
不足する場合のみ、再現手順・期待挙動・対象範囲・優先度などを簡潔に確認する。

### Phase 2: リポジトリを根拠として調査する

提案手法を具体化するため、対象コードを `Glob` / `Grep` / `Read` で特定する。

- バグの場合: 症状が現れる箇所から呼び出し経路をたどり、`根本原因` を実コードで裏付ける
- 機能の場合: 追加・変更すべきファイルと既存の拡張点 (インターフェース、設定) を特定する
- 関連する Issue / PR / コミットがあれば `gh` で参照し、`参考文献` に加える

推測で原因や実装箇所を書かない。コードで確認できた事実のみを根拠にする。

### Phase 3: Issue 本文を生成する

共有テンプレート `~/.claude/skills/template/issue.md` を読み込み、各プレースホルダを
Phase 1〜2 の結果で埋める。テンプレートの構成は次のとおり:

| セクション | 書く内容 |
|---|---|
| 概要 | 何を解決するかを 1〜2 文で。タイトルだけで要点が伝わる粒度 |
| 背景 | 起票の動機となった状況 |
| 課題 | 現状の具体的な問題 (再現手順・期待挙動・実際の挙動など) |
| 根本原因 (バグのみ) | 症状でなく原因。ファイルパスと該当箇所で裏付ける |
| ユーザーストーリー (機能のみ) | 「[役割] として [目的] のために [機能] が欲しい」 |
| 目標 | 背景・課題に対する解決後の理想像 |
| 提案手法 | ファイルパス付きコード差分と、代替案との比較・採用理由 |
| 検証方法 | 提案手法が正しく機能するかの確かめ方 |
| 受入基準 | Close 条件。計測可能なチェックボックス箇条書き |
| 作業単位 | 着手可能な単位への分割と依存関係 |
| 補足 | 前提・制約・参考情報。なければ省略 |
| 用語集 | 本文の専門用語・略語を簡潔な説明と一次情報リンクで補足。平易な語のみなら省略 |
| 参考文献 | 関連 Issue / PR / ドキュメント / 仕様へのリンク |

バグ報告なら `ユーザーストーリー` を、機能開発なら `根本原因` を削除する。
起票前にテンプレート冒頭の HTML コメントを削除する。

### セクションごとのガイドライン

- **概要**: タイトル的に要点を 1〜2 文で。詳細は後続セクションに委ねる
- **根本原因 (バグのみ)**: 症状ではなく原因。コードで裏付けられない場合は「調査中」と明示する
- **ユーザーストーリー (機能のみ)**:「[役割]として[目的]のために[機能]が欲しい」の形を守る
- **提案手法**: 必ずファイルパスと具体的なコード差分を含める。抽象論で終わらせない
- **受入基準**: すべて計測可能・検証可能にする。「正しく動く」のような曖昧な表現を避ける
- **作業単位**: 依存関係を必ず示し、並行可能な作業を区別する
- **用語集**: 本文の専門用語・略語・ドメイン固有の語彙を簡潔に説明する。
  各項目に一次情報 (公式ドキュメント・仕様・Wikipedia など) へのリンクを添える。
  平易な語しか出てこない場合は省略してよい

### Issue の作成

本文を生成したら、ユーザーに確認のうえ `gh issue create` での起票を提案する。

```bash
gh issue create --title "<タイトル>" --body-file <生成した本文>
```

ユーザーが本文の生成のみを求めている場合は、起票せず本文を提示するにとどめる。

## Prohibited Actions

- リポジトリを調査せず、推測で根本原因や提案手法を書く
- 提案手法をファイルパス・コードを伴わない抽象論で済ませる
- 受入基準に計測不能な表現 (「正しく動く」「問題なく動作する」) を使う
- バグ報告に `ユーザーストーリー` を、機能開発に `根本原因` を混在させる
- 確認できない事実を断定する。不明な点は「不明」「調査中」と明示する
